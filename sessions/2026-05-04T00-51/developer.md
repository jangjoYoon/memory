## 1. 기술 아키텍처 우선순위 재설정 및 예외 처리 로직 계획

확정된 시스템 명세(Tier 2 상품 기반)와 데이터 파이프라인 안정화 목표를 바탕으로, 핵심 기능 구현을 위한 기술 아키텍처의 우선순위를 재설정하고 예외 처리 로직 구현 계획을 수립합니다.

### 1.1. 기술 아키텍처 우선순위 (Data Flow & Core Features)

시스템 판매 및 신뢰도 확보를 최우선 목표로 설정하여, 데이터 흐름은 **'신뢰성(Reliability)'**과 **'확장성(Scalability)'**을 중심으로 설계합니다.

| 우선순위 | 모듈/영역 | 핵심 목표 | 기술적 결정 사항 | 이유 (수익화 관점) |
| :---: | :--- | :--- | :--- | :--- |
| **P1** | **데이터 수집 및 정제 파이프라인 (Ingestion & Cleaning)** | 원천 데이터의 무결성 100% 보장 | 모든 외부 API 호출에 대한 **멱등성(Idempotency) 확보** 및 실패 시 재시도 로직(Retry Mechanism) 구현. | 데이터 오류는 시스템 신뢰도를 즉각적으로 저해하므로, 가장 먼저 안정화해야 한다. |
| **P2** | **핵심 AI 추론 엔진 (Core Inference Engine)** | 사용자 입력에 대한 정확하고 일관된 결과 도출 | 프롬프트 엔지니어링 모듈을 API 레이어에서 분리하여 관리하고, 버전 관리를 통해 추론 로직의 변경 이력을 명확히 함. | 시스템의 핵심 가치(AI 결과물)가 안정적으로 제공되어야 고객이 비용을 지불한다. |
| **P3** | **결과물 포맷팅 및 배포 (Output Formatting & Deployment)** | 최종 사용자에게 전달되는 결과물의 표준화 | 다양한 출력 형식(JSON, Markdown 등)에 대한 템플릿 엔진을 구축하여 유연하게 대응하고, 오류 발생 시 대체 포맷으로 자동 전환하는 로직 구현. | 마케팅 및 고객 경험 측면에서 일관된 품질을 제공한다. |

### 1.2. 예외 처리 로직 구현 계획 (Exception Handling Roadmap)

데이터 파이프라인의 안정성을 위해 모든 주요 단계에 대해 **'실패 시 복구(Recovery)'** 메커니즘을 설계하고, 이를 개발 로드맵에 따라 순차적으로 적용합니다.

| 단계 | 목표 기능 | 예외 처리 전략 | 구현 세부 사항 | 연관된 개발 로드맵 |
| :---: | :--- | :--- | :--- | :--- |
| **Phase 1** | **API/외부 호출 실패 처리** | **재시도 및 알림 (Retry & Alert)** | HTTP 429(Rate Limit), 5xx 에러 발생 시 지수 백오프(Exponential Backoff) 전략을 적용하여 자동 재시도. 실패 시 Slack/이메일로 즉시 알림 발송. | P1 (데이터 수집) 안정화 |
| **Phase 2** | **데이터 정제 오류 처리** | **격리 및 대체 (Isolate & Substitute)** | 데이터 유효성 검사(Validation) 단계에서 실패한 레코드를 별도의 'Dead Letter Queue (DLQ)'에 저장하고, 시스템은 해당 데이터를 건너뛰고 다음 작업으로 진행. | P1 (데이터 수집) 안정화 |
| **Phase 3** | **AI 추론 엔진 오류 처리** | **프롬프트 안전장치 (Prompt Guardrails)** | LLM 호출 시 발생할 수 있는 환각(Hallucination) 또는 부적절한 응답에 대해 사전에 정의된 안전 프롬프트(System Prompt)를 통해 응답을 필터링하고, 실패 시 사용자에게 명확한 오류 메시지 반환. | P2 (추론 엔진) 구현 |
