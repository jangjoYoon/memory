*   **스타일:** '엔진 해부' 컨셉을 유지하며, 기술 흐름(Input → Process → Output)이 명확하게 분리되고 동기화되어야 합니다.

    3.  **Core Input:** (실제 투입되는 핵심 요소)

#### B. Process 단계 (Process Phase - The Engine)
*   **목표:** 데이터가 처리되고 가공되어 수익 경로로 전환되는 내부 시스템의 작동 원리 해부.
*   **시각화 요소:** 복잡한 기술 흐름(Flowchart)을 동적으로 애니메이션화하여, 각 모듈 간의 상호작용과 변환 과정을 명확히 보여줍니다.
*   **필수 데이터 포인트 (Motion Graphic에 포함):**
    1.  **System Module 1 (Analysis/Deconstruction):** Input 데이터를 분석하는 과정.
    2.  **System Module 2 (Blueprint Application):** 'The Blueprint'를 적용하여 로드맵을 설정하는 과정.
    3.  **System Module 3 (Profit Path Calculation):** 수익 경로(Profit Path)가 계산되고 최적화되는 과정.
    4.  **Optimization Loop:** 반복적인 피드백 및 개선 루프 강조.

#### C. Output 단계 (Output Phase - The Result)
*   **목표:** 최종적으로 도출된 결과물과 시청자에게 전달될 가치(수익)를 명확히 제시.
*   **시각화 요소:** 계산된 수익 경로가 최종적인 결과물로 명확하게 분리되어 강조됩니다.
*   **필수 데이터 포인트 (Motion Graphic에 포함):**
    1.  **Final Output:** 도출된 최종 상품/시스템(예: AI 기반 파이프라인 설계 템플릿).
    2.  **Quantified Result:** 예상되는 수익 경로의 시각적 표현 (오렌지색으로 최대 강조).
    3.  **System Authority:** '시스템 설계자'로서의 권위가 최종 결과물에 투영됨을 암시하는 마무리 컷.

### 3. 기술 구현 요구사항 (Technical Implementation Checklist)

1.  **Flowchart 동기화:** Input, Process, Output 간의 전환은 **오렌지색 선(FF8C00)**으로 명확하게 연결되어야 하며, 각 단계는 시간 순서대로 논리적으로 흐름을 따라야 합니다.

3.  **데이터 레이블링:** 모든 주요 데이터 포인트와 모듈에는 명확하고 간결한 텍스트 레이블을 삽입하여 시각적 정보 전달의 정확성을 확보합니다. (추상적인 설명 금지)

## 핵심 Flowchart 시각적 요구사항 설계 (Profit Path 동기화 기반)

Business의 수익 경로(Profit Path) 동기화 검토 결과를 바탕으로, 오렌지색 강조 원칙(`FF8C00`)을 적용한 핵심 Flowchart의 초기 레이아웃 및 애니메이션 흐름에 대한 구체적인 시각적 요구사항을 다음과 같이 설계합니다. 목표는 '시스템 설계자'로서의 권위를 시각적으로 입증하고 실행력을 극대화하는 것입니다.

### 1. 전체 레이아웃 컨셉: '엔진 해부' (Engine Dissection)
Flowchart는 단순한 순서도가 아닌, 복잡한 시스템을 해부하여 수익 경로를 명확히 보여주는 **역동적인 엔진 해부(Engine Dissection)** 컨셉으로 설계합니다.

*   **구조:** 좌측에서 우측으로 흐르는 선형 구조(Linear Flow)를 기본으로 하되, 각 단계는 독립된 모듈처럼 분리되어 시각적 계층을 형성해야 합니다.
*   **흐름 강조:** 전체 흐름은 오렌지색(`FF8C00`)의 두꺼운 화살표와 연결선으로 표현하여, 사용자가 '어디서부터 어디로' 가는지 즉각적으로 인지하게 합니다.

### 2. 색상 및 강조 원칙 적용 (`FF8C00` 활용)
오렌지색은 단순한 강조가 아닌, **핵심 수익 엔진**을 상징하도록 사용합니다.

*   **핵심 경로 (Profit Path):** 수익이 발생하는 핵심 단계(Input → Process → Output)는 오직 `FF8C00`으로 채워진 직사각형 또는 모듈로 표현하여 시각적 권위를 극대화합니다.
*   **시스템 경계:** 일반적인 프로세스나 데이터 입력/출력 단계는 중립적인 색상(예: 짙은 회색 또는 흰색 배경)을 사용하여, 오렌지색 경로가 시스템의 핵심임을 부각시킵니다.
*   **결과 강조:** 최종 결과물(Output) 단계는 가장 밝고 강렬한 오렌지색으로 처리하여 시청자의 최종 목표를 명확히 제시합니다.

### 3. 애니메이션 흐름 설계 (Motion Graphic Flow)
애니메이션은 정적인 정보 전달이 아닌, **시스템의 작동 과정**을 보여주는 동적인 해부 과정을 따라야 합니다.

*   **시작점 (Input):** 데이터나 자원이 시스템으로 유입되는 순간을 강조합니다. 입력 단계는 긴장감 있게 시작하며, 오렌지색 화살표가 이 지점을 강력하게 가리킵니다.
*   **처리 과정 (Process):** 각 프로세스 모듈이 활성화될 때마다 내부에서 데이터나 흐름이 움직이는 애니메이션(예: 데이터 스트림의 이동)을 삽입하여 '해부'의 느낌을 부여합니다. 이 과정은 오렌지색으로 연결되어야 합니다.
*   **결과 도출 (Output):** 최종 결과물이 도출되는 순간, 화면 전체에 `FF8C00` 색상이 폭발적으로 나타나며 명확한 성공 신호(Success Signal)를 제공합니다.

## 핵심 Flowchart 시각적 요구사항 재확인 및 보강 (오렌지 강조 원칙 적용)

Business의 수익 경로 동기화 검토 결과를 바탕으로, 오렌지색 강조 원칙(`FF8C00`)이 적용된 핵심 Flowchart의 초기 레이아웃 및 애니메이션 흐름에 대한 구체적인 시각적 요구사항을 다음과 같이 재확인하고 보강합니다. 이 요구사항은 Developer가 Motion Graphic 제작을 즉시 시작할 수 있도록 모듈화하여 제시합니다.

### 1. 전체 구조 및 레이아웃 (Layout & Structure)

**컨셉:** '엔진 해부(Engine Dissection)' 컨셉을 유지하며, 시스템의 논리적 흐름(Input → Process → Output)이 명확하게 시각화되어야 합니다.

*   **레이아웃 형태:** 수평 또는 계단식(Waterfall) 구조를 기본으로 하되, 각 모듈 간의 연결은 화살표가 아닌 **'에너지/데이터 흐름'**을 나타내는 유기적인 곡선이나 흐름선으로 표현하여 역동성을 부여합니다.
*   **모듈 분리:** Flowchart는 최소 3개의 명확한 섹션으로 분리되어야 합니다.
    1.  **Input (Input Layer):** 외부 데이터/자원 유입 단계.
    2.  **Process (Engine Core):** 시스템 설계 및 변환 과정 (가장 복잡하고 핵심적인 부분).
    3.  **Output (Profit Path):** 최종 결과물 및 수익 경로 도출 단계.

### 2. 색상 및 강조 원칙 적용 (Color & Emphasis - `FF8C00`)

오렌지색(`FF8C00`)은 **'가치 창출(Value Creation)'**과 **'수익 발생 지점'**을 시각적으로 압도하는 데 사용되어야 합니다.

*   **기본 흐름 (Default Flow):** 일반적인 시스템의 Input 및 Process 단계는 차분한 중립색(예: 짙은 회색 또는 어두운 네이비)을 사용하여 배경 역할을 수행합니다.
*   **핵심 강조 (Critical Emphasis - `FF8C00` 적용):**
    *   **Input → Process 전환 지점:** 데이터가 시스템에 투입되는 순간, 그리고 핵심 알고리즘이 시작되는 지점에 오렌지색으로 강하게 하이라이트합니다.
    *   **Process 내부의 핵심 모듈:** 수익 경로를 직접적으로 결정하는 **'핵심 변환 단계(Key Transformation Steps)'**는 반드시 `FF8C00` 배경 또는 테두리로 강조하여 시청자의 시선을 즉시 집중시켜야 합니다. (예: '시스템 설계', '데이터 해부', '수익 공식 적용' 등)
    *   **Output 지점:** 최종 수익 경로(Profit Path)가 도출되는 최종 단계는 가장 밝고 강렬한 오렌지색으로 마무리하여 성취감을 극대화합니다.

### 3. 애니메이션 흐름 (Animation Flow - Motion & Dynamics)

애니메이션은 정적인 다이어그램이 아닌, **'시스템 작동 시뮬레이션'**처럼 느껴져야 합니다.

*   **시작:** Input 데이터가 화면에 나타나며 시작됩니다.
*   **흐름:** 데이터가 Process 모듈로 유입될 때, 오렌지색 흐름선(`FF8C00`)이 활성화되며 시스템 내부에서 복잡한 계산(해부)이 일어나는 듯한 역동적인 움직임을 보여줍니다.
*   **핵심 강조 시퀀스:** 가장 중요한 Process 모듈을 통과할 때, 해당 모듈의 경계가 오렌지색으로 '점등'되거나 '폭발'하는 듯한 효과를 사용하여 **"여기서 가치가 창출된다"**는 메시지를 전달합니다.
*   **결론:** 최종 Output 단계에 도달하면, 모든 흐름이 깔끔하게 정리되며 최종 수익 경로가 오렌지색으로 명확하게 표시되어 시청자에게 명쾌한 해답을 제시하며 마무리됩니다.

## ⚙️ Motion Graphic/Flowchart 구현을 위한 기술 요구사항 명세 (Data & API Specification)

본 Flowchart는 **'엔진 해부(Engine Dissection)'** 컨셉을 시각화하며, Input → Process → Output의 데이터 흐름을 중심으로 설계됩니다. 모든 시각적 요소는 아래 정의된 JSON 구조를 기반으로 동적으로 생성되어야 합니다.

### 1. 핵심 데이터 구조 (JSON Schema Definition)

모든 Flowchart 노드(Input, Process, Output)는 다음의 표준화된 JSON 객체 구조를 따라야 합니다. 오렌지색 강조 원칙(`FF8C00`)은 `type` 필드를 통해 동적으로 적용됩니다.

```json
{
  "node_id": "unique_uuid_string",
  "type": "Input" | "Process" | "Output", // 노드의 종류 (색상 및 레이아웃 결정)
  "title": "노드에 표시될 핵심 제목",
  "description": "해당 단계의 상세 설명 (Motion Graphic 텍스트)",
  "data_source": "API_ENDPOINT_OR_DATASET_ID", // 이 노드가 참조하는 데이터 소스 (API 연결 지점)
  "dependencies": ["node_id_1", "node_id_2"], // 선행되어야 할 노드 ID 목록
  "transformation_logic": { // Process 노드에만 적용되는 핵심 로직 명세
    "operation": "calculate" | "transform" | "filter",
    "formula_or_logic": "사용될 구체적인 수학적/논리적 공식 또는 알고리즘 설명",
    "input_fields": ["field_A", "field_B"], // 입력으로 사용되는 데이터 필드 명세
    "output_fields": ["result_X"] // 결과로 도출되는 데이터 필드 명세
  },
  "visual_emphasis": { // Designer의 색상 규칙 적용
    "highlight_color": "FF8C00" | "Neutral", // 강조 색상 설정
    "style": "Waterfall" | "EngineDissection" // 레이아웃 스타일 지정
  }
}
```

### 2. 데이터 파이프라인 및 API 연결 요구사항 (Data Flow & API Integration)

Motion Graphic의 동적 흐름을 위해, 각 `Process` 노드는 외부 데이터 소스(`data_source`)로부터 데이터를 받아와야 합니다.

1.  **Input Layer (데이터 수집):**
    *   **요구사항:** 시스템이 시작될 때 필요한 초기 입력 데이터(예: 시장 데이터, 사용자 설정값)를 Mock-up 또는 실제 API 호출을 통해 확보해야 합니다.
    *   **API 명세 예시:** `GET /api/v1/initial_data?context=profit_path`

2.  **Engine Core (데이터 변환):**
    *   **요구사항:** 각 `Process` 노드는 이전 단계의 출력 데이터(`dependencies`를 통해 참조)와 외부에서 수집한 데이터를 결합하여 `transformation_logic`에 명시된 로직을 실행해야 합니다.
    *   **API/Logic 통합:** 복잡한 계산(예: 수익 공식 적용)은 백엔드 API 호출 대신, Motion Graphic 내에서 **'가상 알고리즘 시뮬레이션'** 형태로 표현되어야 하며, 이 시뮬레이션 과정이 `transformation_logic`에 의해 명확히 정의되어야 합니다.

3.  **Output Layer (결과 도출):**
    *   **요구사항:** 최종 결과물(`Output`)은 모든 변환 과정을 거친 후의 최종 값이어야 하며, 이 값이 시각적으로 가장 강조되어야 합니다.
    *   **API 명세 예시:** `POST /api/v1/profit_path/final_result` (최종 수익 경로를 도출하는 엔드포인트)
